JavaScript Engine Internals

Code Execution in JavaScript

Every script runs in two distinct phases inside the JS Engine. Understanding this explains hoisting, undefined values, and how bugs appear before your code even "runs".

Phase 1

Memory Creation Phase

  • JS Engine scans the entire code first
  • Allocates memory for all variables & functions
  • var variables → initialised as undefined
  • let / const → enter Temporal Dead Zone
→
Phase 2

Code Execution Phase

  • Engine executes code line by line
  • Actual values are assigned to variables
  • Data types are determined dynamically
  • Functions run when invoked
1

Script Loaded — Execution Context Created

The browser loads script.js. The JS Engine creates a Global Execution Context and immediately enters the Memory Creation Phase — before a single line of code runs.

The debugger is open but not yet started. No breakpoint hit yet.
DevTools → Sources → Debugger not yet paused
2

Breakpoint Set on Line 2

We set a breakpoint at console.log("myUniqueName"). When we reload, execution pauses before line 2 runs. The Scope panel shows the current state of memory.

"Paused on breakpoint" — execution hasn't started yet.
Paused on breakpoint — Scope panel visible
3

Memory Creation Phase — Hoisting

The JS Engine has scanned all declarations. Look at the Scope → Script panel: myUniqueName exists in memory but its value is undefined. The engine knows the variable exists — but hasn't assigned its value yet.

myUniqueName:undefined
age:<value unavailable>
city:<value unavailable>
This is hoisting in action — memory allocated, no values yet.
Scope panel → myUniqueName: undefined (memory phase)
4

Line 2 Executes — console.log Fires

Code Execution Phase begins. Line 2 (console.log("myUniqueName")) runs. Because we are still in the execution phase and myUniqueName hasn't been assigned yet, the console prints undefined.

This is exactly why hoisting matters — the variable exists, but has no value yet at this point in execution.
Line 2 executes → myUniqueName: undefined in Global scope
5

Line 3 Executes — Value Assigned

var myUniqueName = 'andrew' runs. The JS Engine now writes the actual value "andrew" into the memory slot that was previously undefined. Check the Scope panel — the value has updated.

myUniqueName:"andrew"
age:undefined
city:undefined
Memory slot updated: undefined → "andrew"
After line 3 → myUniqueName: "andrew" in Scope
6

Line 4 Executes — age = 30

const age = 30 runs. The Scope panel now shows age: 30. Notice that city is still unavailable — the engine executes strictly line by line.

myUniqueName:"andrew"
age:30
city:<value unavailable>
After line 4 → age: 30, city still unavailable
7

Line 5 Executes — city = 'New York'

let city = 'New York' runs. All three variables now have their actual values in the Scope panel. The Code Execution Phase is nearly complete.

myUniqueName:"andrew"
age:30
city:"New York"
All values assigned — execution phase complete.
After line 5 → all variables have values
8

Script Ends — Execution Context Removed

The engine reaches the end of the script. The Global Execution Context is popped off the Call Stack and removed from memory. The console shows undefined — the value that was logged back in Step 4, before myUniqueName had its value.

The output "undefined" in the console is proof that console.log ran during execution, before the assignment on line 3.
Console output: undefined — logged before value was assigned

Why undefined?

var declarations are hoisted and initialised as undefined during the Memory Creation Phase. So console.log on line 2 runs before line 3 assigns the value — printing undefined.

var vs let/const

var → hoisted as undefined.
let / const → hoisted into a Temporal Dead Zone. Accessing them before declaration throws a ReferenceError, not undefined.

Key Insight

JavaScript always runs in two passes. The first pass (Memory Phase) is why variables "exist" before their line runs. The second pass (Execution Phase) is where values are actually assigned, line by line.
